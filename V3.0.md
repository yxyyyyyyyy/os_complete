# AORT-R V3.0 历史执行方案

> 本文是阶段性开发提示，不是当前能力清单。当前事实、两条评审主线和禁止宣称项以 `docs/design/README.md` 与 `docs/defense/CLAIMS_BOUNDARY.md` 为准。

# 一、项目定位

AORT-R 不是普通多 Agent Workflow，不是简单聊天机器人，也不是 LangGraph / CrewAI / AutoGen 的套壳。

AORT-R 的定位是：

运行在 openEuler/Linux 上的用户态 Agent Runtime 原型。它不修改 Linux 内核；部分 Linux 能力按环境启用并通过 evidence_mode 标注，eBPF 不是核心路径的必备条件。

核心目标：

1. Agent 是真实 worker 进程，有 PID。
2. Agent 有 AVP Control Block，记录生命周期、状态、资源、上下文页表、调度信息。
3. Agent 进入独立 cgroup v2，支持 memory/pids/cpu 限制和 freeze/kill。
4. Agent 不直接执行工具，必须通过 UDS JSON-RPC 请求 aortd 的 Syscall Gateway。
5. 上下文通过 CVM Context Virtual Memory 管理，拆成内容寻址 Page 和 per-agent PageTable。
6. Agent 间通信尽量传 page_id，而不是复制大文本。
7. 调度器支持 FIFO、token-CFS、token-CFS-prefix-affinity。
8. Runtime 支持动态 agent.spawn，例如 Reviewer 在测试失败后动态创建 Fixer。
9. Runtime 支持 Supervisor 故障隔离，例如 tool timeout、heartbeat lost、pids limit、workspace rollback。
10. Dashboard 必须展示真实 Runtime 证据，而不是只展示 mock 事件。
11. 实验必须尽量基于真实 Runtime 数据，不能伪装模拟结果。

# 二、开发总原则

你必须遵守以下原则：

1. 先检查现有代码，不要推翻重写。
2. 尽量保留当前项目结构，在现有基础上补齐模块。
3. 每个阶段只做该阶段目标，不要偷偷做后面阶段。
4. 所有新增功能必须有：

   * 后端结构体或接口
   * API 或内部调用链
   * trace/event 记录
   * Dashboard 或报告中的可见证据
   * 基础测试或手动验证命令
5. 所有 Linux 特性必须支持 degraded mode：

   * 没有 root 权限时，不要 panic
   * 非 Linux 环境时，不要 panic
   * cgroup v2 不可用时，进入 degraded mode
   * overlayfs 不可用时，fallback 到 copy workspace
6. 不允许伪造真实实验数据。

   * 如果是模拟数据，必须标记 mode=simulation
   * 如果是半真实数据，必须标记 mode=degraded-real
   * 如果是真实运行数据，才能标记 mode=real
7. 每个阶段结束必须运行能运行的测试或验证命令。
8. 每个阶段结束必须生成阶段报告文档。
9. 阶段报告写完后必须停止，不要继续下一阶段。
10. 不要把 node_modules、dist、.git、.cache、__MACOSX、Go build cache、日志文件放进最终提交包。

# 三、每阶段必须输出的报告文档

每完成一个阶段，必须创建：

```text

```

```text
docs/phase_reports/PHASE_0_REPORT.md
docs/phase_reports/PHASE_1_REPORT.md
docs/phase_reports/PHASE_2_REPORT.md
```

每份报告必须包含以下内容：

````markdown
# PHASE X REPORT

## 1. 阶段目标

说明本阶段原计划做什么。

## 2. 本阶段实际完成内容

逐条列出已经实现的功能。

## 3. 修改文件清单

列出所有新增/修改/删除的文件，并说明每个文件的作用。

格式：

| 文件 | 类型 | 说明 |
|---|---|---|
| internal/xxx/xxx.go | 修改 | xxx |
| cmd/xxx/main.go | 新增 | xxx |

## 4. 核心实现说明

说明关键结构体、关键函数、核心流程。

必须写清楚：

- 入口在哪里
- 数据怎么流动
- Runtime 怎么记录 trace/event
- Dashboard 或 API 如何看到结果

## 5. 新增或修改的 API

如果有 API，列出：

| 方法 | 路径 | 作用 | 返回关键字段 |
|---|---|---|---|

如果没有 API，写“本阶段无新增 API”。

## 6. 验证命令

列出实际运行过或建议运行的命令。

例如：

```bash
go test ./...
go run ./cmd/aortd
````

## 7. 验证结果

必须如实写：

* 哪些命令通过了
* 哪些命令失败了
* 失败原因是什么
* 有没有 degraded mode

不能假装测试通过。

## 8. 当前风险和遗留问题

列出本阶段还没解决的问题。

## 9. 下一阶段建议

写下一阶段应该做什么，但不要直接开始做。


# PHASE 1：真实 Worker 进程与 UDS 注册

## 阶段目标

把项目从 mock demo 往真实 Runtime 推进。
aortd 必须能启动真实 Agent Worker 进程。
Worker 必须通过 Unix Domain Socket 注册并发送 heartbeat。

## 你要做的事情

1. 检查现有 `cmd/aort-worker`、`internal/worker`、`internal/api`。
2. 如果已有 worker，完善它；如果不完整，补齐。
3. aortd 在 demo run 中启动真实 worker 进程。
4. worker 启动参数至少包含：

   * agent-id
   * role
   * task-id
   * sock path
5. worker 启动后通过 UDS 向 aortd 注册。
6. 注册消息至少包含：

   * type=register
   * agent_id
   * role
   * task_id
   * pid
7. aortd 收到注册后更新 AVP：

   * pid
   * state
   * updated_at
8. worker 每 2 秒发送 heartbeat。
9. aortd 如果 6 秒没有收到 heartbeat，把 Agent 标记为 FAILED。
10. 所有事件必须进入：

* trace recorder
* SSE events

11. 修改 `/api/agents`，确保能看到：

* agent_id
* role
* state
* pid
* last_seen

12. 保留 mock fallback，但默认 demo 应优先走 worker mode。
13. 生成：

```text
docs/phase_reports/PHASE_1_REPORT.md
```

## 阶段结束标准

必须能说明：

* demo 是否真的启动 worker
* worker PID 是否能在 `/api/agents` 看到
* kill worker 后是否能变 FAILED
* 哪些地方仍是 mock

不要进入 PHASE 2。

11. 所有 capsule 操作写 trace 和 SSE event。


# PHASE 3：真实软件工程 Demo

不要只执行 `go version`。

## 你要做的事情

   * 通过 `tool.exec` 执行 `go test ./...`。

   * 读取测试结果。

   * 修复失败代码。
   * 再触发 Tester 或报告修复结果。
7. Timeline 必须出现：

   * worker registered
   * context.materialize
   * context.write_delta
   * tool.exec
   * agent.report
8. Dashboard 必须看到这些 Agent 的真实 PID。
9. 保留 mock fallback，但默认演示走真实 worker mode。

```text
docs/phase_reports/PHASE_3_REPORT.md
```

# PHASE 5：Syscall Gateway 强化

## 阶段目标

让 Worker 不能直接执行工具，必须通过 Runtime 的 syscall gateway。
这部分是“统一模型调用、工具调用、上下文、通信和资源”的核心证据。

## 你要做的事情

1. 检查并完善 `internal/syscall`。
2. Worker 通过 UDS JSON-RPC 请求 aortd 执行能力。
3. 支持 syscall：

   * context.materialize
   * context.write_delta
   * tool.exec
   * agent.report
4. 每个 syscall 记录 SyscallRecord：

   * id
   * agent_id
   * name
   * start_time
   * end_time
   * duration_ms
   * status
   * error
   * input_size
   * output_size
5. `tool.exec` 必须支持 timeout。
6. `tool.exec` 的 cwd 必须限制在 Agent workspace。
7. `tool.exec` 返回 stdout、stderr、exit_code。
8. Timeline 显示：

   * syscall.started
   * syscall.finished
9. API：

   * `GET /api/syscalls`
10. 如果工具超时，必须产生 TOOL_TIMEOUT 错误。
11. 生成：

```text
docs/phase_reports/PHASE_5_REPORT.md
```

## 阶段结束标准

必须能说明：

* Worker 是否真的通过 syscall 调工具
* tool timeout 是否被捕获
* Timeline 如何证明 syscall 发生
* tool.exec 是否限制在 workspace

不要进入 PHASE 6。

# PHASE 6：token-CFS 调度器与调度日志

## 阶段目标

补齐比赛要求里的“统一调度”和“资源感知调度”。

## 你要做的事情

1. 检查并完善 `internal/scheduler`。
2. 支持策略：

   * FIFO
   * token-CFS
   * token-CFS-prefix-affinity
3. AVP 中要有：

   * weight
   * vruntime
   * consumed_tokens
4. token-CFS 规则：

   ```text
   vruntime += consumed_tokens / weight
   ```
5. 每次选择 READY Agent 时，选择 vruntime 最小者。
6. prefix affinity 规则：

   * 如果候选 Agent 与上一个被调度 Agent 共享 project/task page；
   * 且 vruntime 差距小于阈值 K；
   * 优先共享 page 更多的 Agent。
7. 每次调度生成 DecisionLog：

   * candidates
   * selected_agent
   * policy
   * reason
   * vruntime_before
   * vruntime_after
   * shared_pages
   * created_at
8. API：

   * `GET /api/scheduler/decisions`
   * `POST /api/scheduler/policy`
9. Dashboard 展示 DecisionLog。
10. 添加测试：

* FIFO 顺序
* token-CFS 公平性
* prefix affinity 生效

11. 生成：

```text
docs/phase_reports/PHASE_6_REPORT.md
```

## 阶段结束标准

必须能说明：

* 当前调度策略是什么
* 一个 Agent 为什么被选中
* vruntime 如何变化
* prefix affinity 是否真的生效

不要进入 PHASE 7。
