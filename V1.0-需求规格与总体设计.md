版本：V1.0  
日期：2026-07-04  
主平台：openEuler  
项目名：AORT, Agent-Oriented Runtime on openEuler  
中文名：面向多智能体的操作系统级运行时

## 1. 项目定位

AORT 不是普通多 Agent 应用，也不是直接修改 Linux 内核的新操作系统。AORT 的定位是运行在 openEuler 上的用户态 Agent 微内核控制平面。

它将 Agent 抽象为类似操作系统进程的一等执行单元，通过统一的生命周期管理、上下文虚拟内存、Agent 系统调用、运行时中断、故障隔离、调度和可回放轨迹，实现多 Agent 任务的稳定、可控、可观测执行。

一句话描述：

> AORT 在 openEuler 用户态实现一个 Agent 微内核，将 Agent 作为虚拟进程管理，提供上下文虚拟内存、Agent 系统调用、资源感知调度、运行时中断、故障监督和可回放执行轨迹。

## 2. 赛题理解

赛题要求基于 openEuler、openKylin、OpenHarmony 等至少一个国内主流开源操作系统开发，并鼓励在更多 Linux 发行版上编译、运行和测试。

本项目选择 openEuler 作为主平台，原因是：

1. openEuler 是国内主流开源 Linux 发行版，符合赛题平台要求。
2. openEuler 适合展示 Linux OS 能力，例如 systemd、procfs、cgroup、进程模型、资源监控和用户态守护进程。
3. 团队时间有限，openEuler 比 OpenHarmony 更适合快速做出可运行、可演示、可量化的系统。
4. AORT 的核心创新在 Agent Runtime 控制平面，不依赖特定桌面环境或移动端系统能力。

兼容性加分目标：

1. openEuler 作为主提交与主演示环境。
2. openKylin 或 Ubuntu 作为额外 smoke test 环境。
3. OpenHarmony 作为后续适配方向，不进入 V1.0 主线。

## 3. 设计目标

### 3.1 功能目标

V1.0 必须实现：

1. Go 版 `aortd` Runtime Daemon。
2. Web Dashboard。
3. DeepSeek 中转 GPT 的 Responses API 兼容模型调用。
4. Agent Virtual Process, AVP。
5. Context Virtual Memory, CVM。
6. Agent Syscall Layer。
7. Runtime Interrupts。
8. Replayable Agent Trace。
9. DAG Scheduler, 包含 FIFO 与 Resource-Aware 两种策略。
10. Tool capability、权限、超时和审计。
11. Fault Supervisor, 支持 timeout、retry、circuit breaker 与故障注入。
12. 多 Agent 自动软件工程 Demo。
13. openEuler 编译、部署、运行说明。
14. 性能与稳定性实验结果。

### 3.2 创新目标

V1.0 不以“多个 Agent 协作”为创新点，而以“Agent OS Runtime 抽象”为创新点。

核心创新组合：

1. Agent Virtual Process：把 Agent 抽象为虚拟进程。
2. Context Virtual Memory：把多 Agent 上下文抽象为虚拟内存页。
3. Agent Syscall Layer：把工具、LLM、IPC、上下文访问抽象为系统调用。
4. Runtime Interrupts：为 Agent 引入运行时中断与信号机制。
5. Replayable Agent Trace：为多 Agent 执行提供可回放轨迹。

### 3.3 非目标

V1.0 明确不做：

1. 不修改 Linux 内核。
2. 不从零实现容器。
3. 不做完整 MCP 协议兼容。
4. 不实现多机分布式调度。
5. 不实现真正 vLLM KV Cache 底层控制。
6. 不主攻 OpenHarmony 适配。
7. 不训练或微调大模型。
8. 不做大而全的 AI OS 桌面系统。

这些内容可写入未来工作，但不影响 V1.0 主线。

## 4. 总体架构

AORT 采用 Go 用户态 Runtime Daemon + Web Dashboard + Agent Worker 的架构。

```text
                         Web Dashboard
        DAG / AVP 状态 / Context Page / Syscall / Trace / Fault
                               |
                        REST + WebSocket
                               |
                             aortd
------------------------------------------------------------------
| AVP Manager              Agent Virtual Process 管理              |
| CVM Manager              Context Virtual Memory 管理              |
| Syscall Gateway          Agent 系统调用入口                       |
| Scheduler                FIFO / Resource-Aware 调度               |
| Interrupt Controller     中断产生、派发、处理                     |
| Fault Supervisor         超时、重试、熔断、故障恢复               |
| Trace Recorder           可回放执行轨迹                           |
| LLM Client               Responses API 兼容调用                   |
| Observability            procfs/cgroup/运行时指标                 |
------------------------------------------------------------------
                               |
                    Agent Worker Processes
        Planner / Coder / Tester / Reviewer / Fixer / Reporter
                               |
          file / shell / go test / git diff / http / llm tools
```

核心原则：

1. `aortd` 是系统主体，不是普通后端 API。
2. Agent 不直接访问工具、文件、LLM 或其他 Agent，必须通过 AORT syscall。
3. Dashboard 只展示和控制，不承载核心调度逻辑。
4. 所有关键行为写入 trace，以支持调试、展示、实验和答辩回放。

## 5. 核心创新设计

### 5.1 Agent Virtual Process

Agent Virtual Process, AVP, 是 AORT 对 Agent 的核心抽象。它类似操作系统进程，但面向多智能体任务。

AVP 由以下部分组成：

```text
AVP = ACB + Context Page Table + Capability Table + Resource Capsule + Trace Handle
```

对应关系：

```text
OS Process         AORT AVP
PCB                Agent Control Block
Page Table         Context Page Table
File Descriptor    Capability Table
Signal             Runtime Interrupt
cgroup             Resource Capsule
trace/log          Replayable Agent Trace
```

建议数据结构：

```go
type AgentVirtualProcess struct {
    AgentID          string
    TaskID           string
    Role             string
    State            AgentState
    Priority         int
    ParentAgentID    string
    ChildAgentIDs    []string
    Dependencies     []string
    ContextPageTable []ContextPageRef
    Capabilities     []Capability
    ResourceCapsule   ResourceCapsule
    RetryPolicy       RetryPolicy
    TraceID           string
    CreatedAt         time.Time
    StartedAt         time.Time
    FinishedAt        time.Time
}
```

Agent 状态机：

```text
CREATED
READY
RUNNING
WAITING_LLM
WAITING_TOOL
WAITING_IPC
SUSPENDED
COMPLETED
FAILED
KILLED
```

关键状态流转：

```text
CREATED -> READY -> RUNNING
RUNNING -> WAITING_LLM -> READY
RUNNING -> WAITING_TOOL -> READY
RUNNING -> WAITING_IPC -> READY
RUNNING -> COMPLETED
RUNNING -> FAILED
FAILED -> READY      retry
FAILED -> KILLED     exceed retry / fatal
```

Dashboard 需要展示 AVP 列表、状态、角色、优先级、上下文页表、能力表、重试次数和最近 syscall。

### 5.2 Context Virtual Memory

Context Virtual Memory, CVM, 将 Agent 上下文拆成可共享、可引用、可压缩、可延迟展开的 Context Page。

上下文页示例：

```text
ctx://global/base
ctx://project/source-summary
ctx://task/T001/spec
ctx://agent/planner/delta
ctx://agent/coder/delta
ctx://agent/reviewer/feedback
ctx://compressed/history
```

核心机制：

1. Context Page：上下文最小管理单元。
2. Context Page Table：每个 AVP 持有自己的上下文页表。
3. Shared Page：多个 Agent 共享同一上下文页。
4. Copy-on-Write Context：Agent 修改共享上下文时生成 delta page。
5. Lazy Materialization：只有在 LLM 调用前才将 page 展开为 prompt。
6. Context Compression：历史上下文超过阈值后压缩为摘要页。
7. Reference Accounting：统计每个 page 被多少 Agent 引用。

LLM 调用前的 materialize 流程：

```text
AVP Context Page Table
  -> load shared pages
  -> load private delta pages
  -> sort by priority and recency
  -> compress if over budget
  -> materialize prompt
  -> call LLM
  -> write output as new delta page
```

V1.0 实现深度：

1. 真实实现 `ctx://` page registry。
2. 真实实现 AVP 到 context pages 的引用关系。
3. 真实实现 prompt materialization。
4. 真实统计引用次数、重复字符数和 token 估算。
5. 不实现底层 KV Cache 控制。

### 5.3 Agent Syscall Layer

Agent Syscall Layer 将 Agent 对外部能力的访问统一为系统调用。

Agent 不直接调用工具，而是向 `aortd` 发起 syscall：

```text
aort.context.read
aort.context.write
aort.context.materialize
aort.llm.call
aort.tool.exec
aort.file.read
aort.file.write
aort.git.diff
aort.test.run
aort.ipc.send
aort.agent.spawn
aort.agent.kill
```

每次 syscall 经过统一管线：

```text
Syscall Request
  -> Capability Check
  -> Quota Check
  -> Timeout Setup
  -> Audit Start
  -> Execute
  -> Fault Hook
  -> Audit Finish
  -> Trace Record
```

syscall 审计字段：

```text
syscall_id
trace_id
agent_id
task_id
syscall_name
input_summary
permission_result
started_at
finished_at
duration_ms
exit_code
error_type
output_ref
```

V1.0 最小工具能力：

```text
file.read
file.write
shell.exec
git.diff
go.test
http.request
llm.call
```

安全策略：

1. Agent 默认没有工具权限。
2. 每个 AVP 从 Capability Table 中获取可用 syscall。
3. 高风险 syscall 需要显式授权。
4. 所有 syscall 有 timeout。
5. 所有 syscall 写入 audit log 与 trace。

### 5.4 Runtime Interrupts

Runtime Interrupts 为 Agent 引入类似 OS signal 的控制机制。

中断类型：

```text
INT_TOOL_TIMEOUT
INT_TOKEN_BUDGET_LOW
INT_CONTEXT_OVERFLOW
INT_AGENT_FAILED
INT_DEPENDENCY_READY
INT_HUMAN_APPROVAL_REQUIRED
INT_RESOURCE_PRESSURE
INT_RETRY_LIMIT_REACHED
```

中断处理策略：

```text
INT_TOOL_TIMEOUT          kill syscall, mark failed, retry if allowed
INT_TOKEN_BUDGET_LOW      compress context or switch model
INT_CONTEXT_OVERFLOW      trigger context compression
INT_AGENT_FAILED          notify supervisor and scheduler
INT_DEPENDENCY_READY      move waiting AVP to READY
INT_HUMAN_APPROVAL_REQUIRED pause AVP
INT_RESOURCE_PRESSURE     reduce scheduling score
INT_RETRY_LIMIT_REACHED   open circuit breaker
```

中断价值：

1. 将故障、资源压力、依赖变化和人工审批统一为 Runtime 事件。
2. 让 Agent 可暂停、恢复、降级、重试和终止。
3. 让 Dashboard 可以展示系统级控制行为。

### 5.5 Replayable Agent Trace

Replayable Agent Trace 记录多 Agent 系统的全部关键事件，用于可观测性、实验统计、故障排查和答辩演示。

trace 事件类型：

```text
task.created
agent.created
agent.state_changed
scheduler.selected
context.page.created
context.page.mounted
context.materialized
syscall.started
syscall.finished
llm.called
tool.failed
interrupt.raised
supervisor.retry
agent.completed
task.completed
```

trace 字段：

```text
event_id
trace_id
timestamp
event_type
agent_id
task_id
parent_event_id
payload
```

Dashboard 需要支持：

1. 时间线查看。
2. 按 Agent 过滤。
3. 按 syscall 过滤。
4. 按故障事件过滤。
5. 回放一次完整 Demo 执行过程。

## 6. 调度设计

V1.0 实现两种调度器：

1. FIFO Scheduler：作为 baseline。
2. Resource-Aware Scheduler：作为创新与实验重点。

调度输入：

```text
READY AVP list
DAG dependency status
priority
context affinity
resource pressure
retry count
tool availability
token budget
```

Resource-Aware score：

```text
score =
  priority * 10
  + dependency_ready_score
  + context_affinity_score
  - resource_pressure_score
  - retry_penalty
  - tool_wait_penalty
```

调度输出：

```text
selected_agent
score
decision_reason
timestamp
```

Dashboard 必须展示 scheduler decision reason，避免调度机制看起来像固定流程。

## 7. Fault Supervisor 设计

Fault Supervisor 负责隔离单个 Agent 故障，避免级联失败。

监督树：

```text
Runtime Supervisor
 ├── Planner AVP
 ├── Coder AVP
 ├── Tester AVP
 ├── Reviewer AVP
 ├── Fixer AVP
 └── Reporter AVP
```

故障类型：

```text
LLM_ERROR
TOOL_TIMEOUT
TOOL_EXIT_NONZERO
CONTEXT_OVERFLOW
TOKEN_BUDGET_EXCEEDED
AGENT_CRASH
RETRY_LIMIT_REACHED
```

处理策略：

```text
retry
fallback model
context compression
spawn fixer
kill avp
pause task
open circuit breaker
```

V1.0 必须有故障注入 Demo：

1. Tester 首次运行测试失败。
2. Reviewer 通过 syscall 创建 Fixer AVP。
3. Fixer 修复问题。
4. Tester 复测通过。
5. Trace 与 Dashboard 展示失败、重试、恢复全过程。

## 15. 风险与对策

风险 1：被认为只是普通多 Agent 工作流。

对策：

1. PPT 和 Dashboard 主讲 Runtime，不主讲 Agent 角色。
2. 使用 AVP、CVM、Syscall、Interrupt、Trace 五个核心创新抽象。
3. 展示 scheduler decision、context page、syscall audit、trace replay。

风险 2：创新点听起来像概念包装。

对策：

1. 每个创新点必须有真实数据结构。
2. 每个创新点必须有 Dashboard 证据。
3. 每个创新点必须有实验或 Demo 行为证明。

风险 3：两人时间不足。

对策：

1. 不做内核修改、OpenHarmony 主适配、多机调度和完整 MCP。
2. 前后端并行。
3. Demo 使用固定任务集合，保证稳定复现。

风险 4：模型调用不稳定。

对策：

1. LLM Client 支持 mock mode。
2. 固定 Demo prompt 和响应缓存。
3. Dashboard 和实验可基于 replay trace 展示。

风险 5：Resource-Aware 效果不明显。

对策：

1. 不承诺大幅性能提升。
2. 强调可解释调度与稳定性。
3. 使用故障、资源压力、上下文复用场景展示调度差异。

## 17. 答辩话术

如果评委问“创新点是什么”：

> AORT 的创新不是多个 Agent 协作，而是在 openEuler 用户态实现了一个 Agent 微内核。我们把 Agent 抽象为 Agent Virtual Process，用 Context Virtual Memory 管理多 Agent 上下文，用 Agent Syscall Layer 管控工具和系统资源，用 Runtime Interrupts 处理故障和资源压力，用 Replayable Agent Trace 支撑可观测和可复现实验。

如果评委问“为什么是操作系统级”：

> AORT 借鉴 OS 的进程、页表、系统调用、中断、调度、监督和 trace 思想，并在 openEuler 用户态实现为可运行的 Agent Runtime。它不修改内核，但提供面向 Agent 的 OS 级控制平面。

如果评委问“为什么不是普通工作流”：

> 普通工作流只定义步骤顺序。AORT 管理的是 Agent 的生命周期、上下文页表、能力表、系统调用、资源预算、中断、故障恢复和可回放轨迹。Reviewer 可以动态创建 Fixer，Scheduler 会根据资源压力和上下文亲和度做决策，系统不是静态链路。
